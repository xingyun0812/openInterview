# 统一契约基线 v1（2026 年 03 月）

**版本**：v1.0-contract-baseline  
**状态**：已锁定  
**生效日期**：2026-03-27  
**生效范围**：候选人筛选链路、面试智能辅助、导出链路、质量发布门禁  
**合并策略**：冻结中，满足"可解冻条件"前相关分支禁止合并

## 1. RFC 决议记录

### RFC-1：`ai_resume_screen_result` 增加 `screen_status`

- **决议**：在主表增加 `screen_status` 字段，语义与 `parse_status` 一致
- **否决方案**：新增统一任务表（过重）、空值推断（不可维护）
- **演进预留**：V1.1 可抽取统一 `ai_task_record`

### RFC-2：`export_type` 枚举统一

- **决议**：`0=SCREENING_EXCEL`、`1=INTERVIEW_EXCEL`、`2=INTERVIEW_WORD`
- **否决方案**：`1/2/3` 起始（改动大）、纯字符串（留后续）
- **附加约束**：接口统一返回 `code + label`

### RFC-3：MQ / Webhook 命名分层

- **决议**：内部 MQ 保持动作语义，外部 Webhook 使用结果语义，维护映射表
- **否决方案**：内外同名（丢失语义边界）、版本化重命名（过重）

### RFC-4：`ai_question_generate_record` 扩字段

- **决议**：主表直接增加 `interview_id`、`resume_section_id`、`input_snapshot_hash`
- **否决方案**：子表（多 JOIN）、塞 JSON（不可索引）

## 2. 字段字典

### 2.1 `ai_resume_screen_result`（RFC-1）

| 字段 | 类型 | 说明 | 约束 |
|---|---|---|---|
| `screen_status` | tinyint | 1处理中 2成功 3失败 4已取消 | NOT NULL DEFAULT 1 |

- `screen_status=2` 时 `match_score`/`recommend_level` 必须有值
- `screen_status=1` 时 `match_score`/`recommend_level` 允许 NULL

### 2.2 `ai_question_generate_record`（RFC-4）

| 字段 | 类型 | 说明 | 约束 |
|---|---|---|---|
| `interview_id` | bigint | 面试归属 | NOT NULL |
| `resume_section_id` | varchar(64) | 简历片段标识 | NOT NULL |
| `input_snapshot_hash` | varchar(64) | 输入快照哈希 | NOT NULL |

### 2.3 `interview_export_task`（RFC-2）

| 字段 | 类型 | 说明 | 约束 |
|---|---|---|---|
| `job_code` | varchar(64) | 筛选导出场景的岗位编码 | NULL（仅筛选导出必填） |
| `file_hash` | varchar(64) | 导出文件 SHA256 | NULL（完成后回填） |

### 2.4 全链路追踪（日志/消息/审计）

- 必填：`traceId`、`bizCode`、`errorCode`

## 6. 状态机（受 RFC 影响部分）

### 6.1 `screen_status`（筛选任务，与 AI 任务状态机一致）

```
允许：1→2, 1→3, 1→4, 3→1（重试）
禁止：2→任意, 4→任意
```

### 6.2 `review_status`（题目审核）

```
允许：1→2, 1→3
禁止：2→1, 2→3, 3→1, 3→2
```

### 6.3 `task_status`（导出任务）

```
允许：1→2, 1→3, 3→1（仅人工补偿）
禁止：2→1, 2→3, 3→2
```

## 7. 迁移与兼容策略

| 阶段 | 动作 | 约束 |
|---|---|---|
| Phase 0 | 冻结合并 | 当前阶段 |
| Phase 1 | 增量 DDL（加字段/加索引，不删不改旧字段） | 仅 ALTER ADD |
| Phase 2 | 应用兼容发布（写：新字段必写+旧字段双写；读：优先新字段） | 双写+兼容读 |
| Phase 3 | 历史数据回填（`screen_status=2`、assistant 字段从 JSON 回填） | 带审计记录 |
| Phase 4 | 切主读写到新契约 | 旧路径仅保留只读 |
| Phase 5 | 稳定观察后清理兼容逻辑（不做破坏性删表） | 两个发布窗口后 |

## 8. 可解冻条件

1. RFC-1~RFC-4 评审通过并记录（本文档即记录）
2. 本"统一契约基线 v1"发布并公告生效时间
3. 迁移脚本与回滚脚本完成 dry-run
4. 契约回归与新增用例全部通过
5. 事件映射联调通过（MQ 与 Webhook）
6. 追踪字段抽检通过（traceId / bizCode / errorCode）
7. 主控明确下达"解冻"指令
